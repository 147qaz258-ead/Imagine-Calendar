## 最小完整单元说明

**核心原子**：用户 → 发现机会 → 认识同频人 → 看到变化

**判断标准**：
- 每个单元能独立工作？✓
- 每个单元能递归生长？✓（用户越多，匹配越精准，讨论越有价值）
- 即使失败也能获得价值？✓（注册失败获得联系方式，圆桌失败获得偏好数据）

**递归生长路径**：
```
1个用户 → 看日历（获得信息价值）
10个用户 → 开始筛选（获得精准价值）
100个用户 → 开圆桌（获得社交价值）
1000个用户 → 数据驱动推荐（获得智能价值）
```

## STORY-001：用户身份认证

### 基本信息

| 项目 | 内容 |
|------|------|
| Story ID | STORY-001 |
| Story 名称 | 用户身份认证 |
| 关联 PRD | `/docs/PRD-v0.md#F1.1` |
| 优先级 | P0 |
| 依赖 | 无 |

### User Story

```
作为 新用户
我想要 用手机号快速注册登录
以便于 开始使用畅选日历
```

### 背景与动机

这是系统的入口。没有用户身份，后续一切功能都无法进行。核心挑战是：
1. 降低注册门槛（只收手机号）
2. 建立信任感（简洁、专业的设计）
3. 数据安全（手机号作为唯一标识）

### 主要使用场景

**场景1：新用户首次访问**
```
用户从分享链接进入首页
→ 看到"看清那些没人告诉你的真相"
→ 点击"新用户注册"
→ 输入手机号
→ 收到验证码
→ 输入验证码
→ 注册成功，跳转引导页
```

**场景2：老用户登录**
```
用户打开网站
→ 点击"手机号登录"
→ 输入手机号
→ 收到验证码
→ 输入验证码
→ 登录成功，跳转日历页
```

**场景3：验证码发送失败**
```
用户点击"获取验证码"
→ 系统发送失败（短信服务异常）
→ 提示"发送失败，请稍后重试"
→ 提供"重新发送"按钮
```

### 状态机

```
┌─────────┐     输入手机号      ┌─────────┐
│  初始   │ ─────────────────→ │ 已输入   │
└─────────┘                    └─────────┘
                                    │
                               点击获取验证码
                                    ↓
                              ┌─────────┐
                              │ 已发送   │ ←── 60秒冷却
                              └─────────┘
                                    │
                               输入验证码
                                    ↓
                              ┌─────────┐
                              │ 验证中   │
                              └─────────┘
                                 /    \
                            验证成功   验证失败
                               /        \
                              ↓          ↓
                        ┌─────────┐  ┌─────────┐
                        │ 已登录   │  │ 错误提示 │
                        └─────────┘  └─────────┘
```

### 验收标准（AC）

**AC-001：手机号格式校验**
- [ ] 输入11位数字时，格式校验通过
- [ ] 输入非11位或非数字，显示"请输入正确的手机号"
- [ ] 输入为空，按钮置灰不可点击

**AC-002：验证码发送**
- [ ] 点击"获取验证码"，按钮变为倒计时（60秒）
- [ ] 验证码5分钟内有效
- [ ] 60秒内不可重复发送
- [ ] 发送失败提示"发送失败，请稍后重试"

**AC-003：验证码校验**
- [ ] 输入正确验证码，登录成功
- [ ] 输入错误验证码，提示"验证码错误"
- [ ] 验证码过期，提示"验证码已过期，请重新获取"

**AC-004：注册登录区分**
- [ ] 新手机号 → 注册流程，跳转引导页
- [ ] 已有手机号 → 登录流程，跳转日历页

**AC-005：数据记录**
- [ ] 注册时间记录到数据库
- [ ] 登录时间更新到数据库
- [ ] 手机号唯一索引

### 边界与异常

| 边界条件 | 处理方式 |
|----------|----------|
| 手机号已存在 | 提示"账号已存在，请直接登录"，切换到登录流程 |
| 验证码错误3次 | 锁定5分钟，提示"错误次数过多，请5分钟后重试" |
| 网络异常 | 提示"网络异常，请检查网络后重试" |
| 短信服务不可用 | 提示"服务暂时不可用，请稍后重试" |

### 接口契约草案

- [ ] 手机号唯一索引（数据库层）
- [ ] 验证码Redis存储（5分钟TTL）
- [ ] 验证码发送限流（同一手机号60秒1次）
- [ ] JWT token生成（7天有效期）
- [ ] HTTPS传输

## STORY-002：用户画像建立

### 基本信息

| 项目 | 内容 |
|------|------|
| Story ID | STORY-002 |
| Story 名称 | 用户画像建立 |
| 关联 PRD | `/docs/PRD-v0.md#F1.2` |
| 优先级 | P0 |
| 依赖 | STORY-001 |

### User Story

```
作为 新注册学生
我想要 快速完成个人资料填写
以便于 获得个性化的招聘推荐
```

### 背景与动机

用户画像是个性化推荐的基础。核心挑战：
1. 降低填写成本（OCR自动识别）
2. 保证数据质量（学校库匹配）
3. 引导偏好设置（13维度）

### 主要使用场景

**场景1：OCR识别成功**
```
用户进入个人资料页
→ 点击"拍照识别学生证"
→ 调用摄像头拍照
→ OCR识别成功
→ 自动填充学校、姓名、学号
→ 用户确认保存
```

**场景2：OCR识别失败**
```
用户拍照学生证
→ OCR识别失败（照片模糊）
→ 提示"识别失败，请手动输入"
→ 用户手动填写学校、专业等
→ 保存
```

**场景3：偏好设置引导**
```
用户完成基础资料
→ 弹窗引导"设置你的偏好，获得更精准推荐"
→ 用户选择跳过或填写
→ 跳转到偏好设置页
```

### 状态机

```
┌─────────┐     拍照      ┌─────────┐
│  初始   │ ────────────→ │ 拍照中   │
└─────────┘               └─────────┘
                               │
                          OCR识别
                               ↓
                         ┌─────────┐
                         │ 识别中   │
                         └─────────┘
                          /        \
                     识别成功      识别失败
                        /            \
                       ↓              ↓
                 ┌─────────┐    ┌─────────┐
                 │ 自动填充 │    │ 手动输入 │
                 └─────────┘    └─────────┘
                        \            /
                         \          /
                          ↓        ↓
                        ┌─────────┐
                        │ 已填写   │
                        └─────────┘
                             │
                          点击保存
                             ↓
                        ┌─────────┐
                        │ 已保存   │
                        └─────────┘
```

### 验收标准（AC）

**AC-001：OCR识别**
- [ ] 调用手机/电脑摄像头
- [ ] 识别学校名称、姓名、学号
- [ ] 准确率≥85%
- [ ] 识别失败时提供手动输入入口

**AC-002：学校匹配**
- [ ] 学校名称自动匹配教育部学校库
- [ ] 不在学校库时提示"请联系客服添加"
- [ ] 学校库包含全国所有高校

**AC-003：基础资料字段**
- [ ] 学校：必填，下拉选择
- [ ] 专业：必填，下拉选择
- [ ] 年级：必填，单选
- [ ] 毕业年份：自动计算
- [ ] 城市：选填，下拉选择

**AC-004：偏好设置**
- [ ] 13个维度可多选
- [ ] 可跳过设置
- [ ] 后续可在设置页修改

### 边界与异常

| 边界条件 | 处理方式 |
|----------|----------|
| 学校不在库中 | 提示"请联系客服添加学校"，允许先选择"其他" |
| 照片模糊 | 提示"照片不清晰，请重新拍摄" |
| OCR服务不可用 | 直接跳转手动填写 |
| 必填字段为空 | 提示"请完成必填项"，保存按钮置灰 |

### 接口契约草案

- [ ] OCR服务接入（百度/阿里云）

- [ ] 偏好数据结构设计（JSON）

## STORY-003：机会发现核心

### 基本信息

| 项目 | 内容 |
|------|------|
| Story ID | STORY-003 |
| Story 名称 | 机会发现核心 |
| 关联 PRD | `/docs/PRD-v0.md#F2` |
| 优先级 | P0 |
| 依赖 | STORY-002 |

### User Story

```
作为 学生
我想要 看到当月所有适合我的招聘活动
以便于 规划我的求职时间
```

### 背景与动机

日历是产品的核心价值入口。解决"信息黑洞"痛点：
1. 聚合所有招聘信息
2. 可视化时间安排
3. 颜色编码快速识别

### 主要使用场景

**场景1：首次访问日历**
```
用户登录后自动跳转日历页
→ 显示当前月份
→ 看到所有招聘活动
→ 颜色区分企业类型
```

**场景2：查看事件详情**
```
用户点击某个事件
→ 弹出事件详情卡片
→ 显示企业、时间、地点、岗位
→ 点击[关注]加入关注列表
```

**场景3：月份切换**
```
用户点击"<"或">"
→ 切换到上/下月
→ 加载该月事件数据
```

### 状态机

```
┌─────────┐
│  初始   │
└─────────┘
     │
  加载数据
     ↓
┌─────────┐
│  加载中  │ ←─── 骨架屏
└─────────┘
     │
  加载完成
     ↓
┌─────────┐     点击事件     ┌─────────┐
│ 正常浏览 │ ─────────────→ │ 详情展示 │
└─────────┘                  └─────────┘
     │                            │
  切换月份                      关闭详情
     ↓                            ↓
┌─────────┐                  ┌─────────┐
│ 重新加载 │                  │ 正常浏览 │
└─────────┘                  └─────────┘
```

- [ ] 单日事件≤3个：全部显示
- [ ] 单日事件>3个：显示"+N更多"
- [ ] 点击"+N更多"展开列表
