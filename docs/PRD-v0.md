## 1. 产品定位与体验北极星

### 1.1 一句话定位

学生职业规划领域的"张雪峰"——帮二本三本学生看清那些没人告诉他的职业真相。

### 1.2 体验北极星（North Star）

> **让学生第一次感受到"原来还有人懂我的迷茫"**

### 1.3 体验原则（用于取舍与争议对齐）

1. **信息要做减法**：学生痛点是信息过载而非信息不足，推送必须精准
2. **认知要可视化**：抽象的成长过程必须变成看得见的图谱
3. **同伴要先于专家**：同频同伴的讨论比专家说教更有价值

---

## 2. 背景与动机

### 2.1 目标用户画像

**核心人群**：二本三本学生，有向上发展意愿但缺乏有效信息、背景、资源

| 特征 | 描述 |
|------|------|
| 年龄 | 18-24岁 |
| 学历 | 二本/三本在校生 |
| 痛点层级 | 信息黑洞 → 认知孤岛 → 路径迷雾 |
| 行为特征 | 刷短视频、微信群活跃、对陌生平台信任度低 |
| 核心诉求 | 找到适合自己的机会、认识同频的人、看清自己的路 |

### 2.2 核心痛点

| 层级 | 痛点 | 表现 | MVP 对应方案 |
|------|------|------|--------------|
| 第一层 | 信息黑洞 | 招聘信息满天飞但不知道哪些适合自己 | 日历 + 13维度筛选 |
| 第二层 | 认知孤岛 | 身边同学都迷茫，聊不出新东西 | 圆桌讨论 + 6人小组 |
| 第三层 | 路径迷雾 | 不知道该补什么、怎么补 | 认知边界图 + 解锁流程 |

## 3. 目标与成功标准

### 3.1 业务目标（引用 biz-overview）

1. 验证学生愿意来（获取）
2. 验证学生愿意用（参与）
3. 验证用了有帮助（效果）

| 指标 | 定义 | 目标 | 埋点口径 |

| 圆桌报名率 | 报名圆桌用户数 / 注册用户数 | ≥90% | 点击报名按钮事件 |

| 次日留存率 | 第2天活跃用户数 / 首日新增用户数 | ≥90% | 次日任意页面访问 |
| 7日留存率 | 第7天活跃用户数 / 首日新增用户数 | ≥90% | 第7日任意页面访问 |

## 5. 功能需求规格

### F1 学生端首页

#### F1.1 注册/登录

**用户故事**：作为新用户，我希望用手机号快速注册，以便开始使用平台

**交互流程**：
```
首页 → 输入手机号 → 获取验证码 → 输入验证码 → 注册成功 → 引导页
```

**字段要求**：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 手机号 | string | 是 | 11位，唯一标识 |
| 验证码 | string | 是 | 6位数字，5分钟有效 |
| 昵称 | string | 否 | 默认"用户+手机后4位" |

**边界条件**：
- 手机号已存在 → 提示"账号已存在，请直接登录"
- 验证码错误 → 提示"验证码错误，请重新输入"
- 验证码过期 → 提示"验证码已过期，请重新获取"

**验收标准**：
- [ ] 手机号格式校验（11位数字）
- [ ] 验证码发送间隔≥60秒
- [ ] 注册成功后自动登录
- [ ] 注册时间记录到数据库

#### F1.2 学生证OCR

**用户故事**：作为学生，我希望拍照自动填充学校信息，以便快速完成资料

**交互流程**：
```
个人资料页 → 点击"拍照识别" → 调用摄像头 → 拍照 → OCR识别 → 自动填充
```

**OCR识别字段**：

| 字段 | OCR字段名 | 说明 |
|------|-----------|------|
| 学校 | school_name | 优先从教育部学校库匹配 |
| 姓名 | name | - |
| 学号 | student_id | - |
| 学院 | college | [OPEN] 是否必填 |

**边界条件**：
- OCR识别失败 → 提示"识别失败，请手动输入"
- 学校不在库中 → 提示"请联系客服添加学校"
- 照片模糊 → 提示"请重新拍摄清晰照片"

**验收标准**：
- [ ] OCR识别准确率≥85%
- [ ] 识别失败时可手动输入
- [ ] 学校名称自动匹配教育部库

#### F1.3 个人资料

**字段列表**：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 学校 | select | 是 | 从学校库选择 |
| 专业 | select | 是 | 从专业库选择 |
| 年级 | select | 是 | 大一/大二/大三/大四/研一/研二/研三 |
| 毕业年份 | select | 是 | 自动计算 |
| 城市 | select | 否 | 当前所在城市 |
| 自我定位 | multi-select | 否 | 13维度之一 |
| 发展方向 | multi-select | 否 | 13维度之一 |

### F3 圆桌讨论

#### F3.1 匹配系统

**用户故事**：作为学生，我希望和相似偏好的人一起讨论，以便产生有价值的话题

**匹配规则**：

| 维度 | 权重 | 说明 |
|------|------|------|
| 学校层次 | 30% | 二本/三本优先匹配 |
| 地点偏好 | 20% | 同城市优先 |
| 行业偏好 | 20% | 同行业优先 |
| 自我定位 | 15% | 同定位优先 |
| 发展方向 | 15% | 同方向优先 |

**匹配流程**：
```
报名 → 填写偏好 → 系统匹配 → 凑齐6人 → 通知会议时间
```

**边界条件**：
- 72小时内凑不齐6人 → 提示"匹配中，请耐心等待"
- 用户取消报名 → 从匹配队列移除
- 会议前2小时 → 发送提醒通知

**验收标准**：
- [ ] 偏好信息收集完整
- [ ] 匹配算法按权重计算
- [ ] 6人小组自动成组
- [ ] 匹配失败有超时处理

#### F3.2 会议流程

**2小时流程设计**：

| 时间 | 环节 | 内容 | 主持人 |
|------|------|------|--------|
| 0-10min | 破冰 | 自我介绍 | 系统引导 |
| 10-40min | 话题讨论1 | 系统推送话题 | 轮值主持 |
| 40-70min | 话题讨论2 | 用户发起话题 | 轮值主持 |
| 70-100min | 认知盘点 | 填写认知图更新 | 个人填写 |
| 100-120min | 下一步约定 | 约定后续行动 | 共同决定 |

**话题来源**：
- 系统预设20个核心问题（[OPEN] 问题清单待确认）
- 用户现场发起

**验收标准**：
- [ ] 会议时间提醒
- [ ] 话题推送机制
- [ ] 认知图更新入口
- [ ] 会议记录保存

## 7. 业务约束与风险

### 7.1 数据安全要求

- 第一批用户原始数据永久保留
- 全量备份：每日凌晨，保留30天
- 增量备份：每小时，保留7天
- 手机号作为唯一标识，数据云端统一存储

### 7.2 技术约束

| 约束 | 说明 |
|------|------|
| 先开发Web端 | 覆盖全部功能 |
| 后续迁移 | 小程序、App |
| 核心风险 | 数据丢失 |

### 7.3 业务风险

| 风险 | 等级 | 应对措施 |
|------|------|----------|
